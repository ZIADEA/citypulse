0.1 Vue d'ensemble des inputs
  L'application reçoit 3 types d'inputs distincts :
  (A) Données importées via fichiers CSV/JSON (clients, véhicules)
  (B) Données saisies manuellement dans l'interface
  (C) Paramètres de simulation choisis par l'utilisateur dans l'interface

  IMPORTANT — Trafic & Météo :
  - La météo et le trafic NE sont PAS des données live/temps réel
  - Ce sont des coefficients multiplicateurs choisis manuellement par l'utilisateur
  - Exemple : l'utilisateur sélectionne "Heure de pointe ×1.7" dans un menu
  - Ce choix est appliqué par l'algorithme pour ajuster les temps de trajet
  - Aucune API externe météo ou trafic n'est nécessaire

0.2 Inputs Type A — Fichiers importés
  Clients (points de livraison) :
  - Source principale : fichier CSV Solomon VRPTW (benchmark académique standard)
  - Colonnes CSV : CUST_NO, XCOORD, YCOORD, DEMAND, READY_TIME, DUE_TIME,
    SERVICE_TIME
  - Import via QFileDialog → prévisualisation 5 lignes → confirmation → insertion SQLite

  Véhicules :
  - Source : fichier CSV créé manuellement ou importé
  - Colonnes : ID, immatriculation, type, capacite_kg, capacite_m3,
    vitesse_kmh, cout_par_km, depot_assigne

0.3 Inputs Type B — Saisie manuelle dans l'interface
  - Coordonnées GPS du dépôt (adresse + latitude + longitude)
  - Informations véhicule (si pas d'import CSV)
  - Instructions spéciales par client
  - Paramètres des algorithmes (max itérations 2-opt, temps CPU OR-Tools)

0.4 Inputs Type C — Paramètres de simulation (choix utilisateur)
  Trafic (radio buttons dans l'interface) :
    → Fluide        × 1.0
    → Chargé        × 1.3
    → Heure de pointe × 1.7
    → Accident      × 2.2
    → Aléatoire     × [1.0 – 2.5]
    → Personnalisé  × [1.0 – 5.0] (saisie libre)

  Météo (radio buttons dans l'interface) :
    → Ensoleillé    × 1.0
    → Pluie légère  × 1.1
    → Pluie forte   × 1.25
    → Neige         × 1.6
    → Verglas       × 2.0

  Coefficient final appliqué aux temps de trajet = trafic × météo
  Exemple : Heure de pointe + Pluie forte = 1.7 × 1.25 = ×2.125

0-BIS.1 Dataset 1 — Solomon VRPTW Benchmark (Principal)
  Source    : http://w.cba.neu.edu/~msolomon/problems.htm
  Kaggle    : https://www.kaggle.com/datasets/masud7866/solomon-vrptw-benchmark
  Licence   : Académique libre — usage autorisé sans restriction
  Format    : CSV / TXT
  Taille    : 56 instances × 100 clients chacune

  Structure des colonnes :
    CUST_NO      → ID client (0 = dépôt)
    XCOORD       → Coordonnée X (longitude simulée)
    YCOORD       → Coordonnée Y (latitude simulée)
    DEMAND       → Demande en unités de charge (kg)
    READY_TIME   → Heure d'ouverture de la fenêtre horaire
    DUE_TIME     → Heure de fermeture de la fenêtre horaire
    SERVICE_TIME → Durée de déchargement au client (minutes)

  Instances recommandées pour les 3 scénarios de démo :
    Scénario 10 clients  → prendre les 10 premières lignes de C101.txt
    Scénario 50 clients  → prendre les 50 premières lignes de R101.txt
    Scénario 100 clients → instance complète RC101.txt (clients clustered+random)

  Catégories disponibles :
    C1xx / C2xx → Clients géographiquement groupés (clustered)
    R1xx / R2xx → Clients répartis aléatoirement (random)
    RC1xx / RC2xx → Mix des deux types

  Instructions d'utilisation dans l'application :
    1. Placer les fichiers .txt dans le dossier assets/demo_data/
    2. Le parser CSV lit les colonnes en ignorant les lignes commentaires (#)
    3. XCOORD / YCOORD sont des coordonnées cartésiennes arbitraires
       → Les convertir en GPS réels (Casablanca par défaut) avec la formule :
         latitude  = 33.5731 + (YCOORD - 50) * 0.01
         longitude = -7.5898 + (XCOORD - 50) * 0.01
    4. DEMAND → utiliser directement comme demande_kg dans la BDD
    5. READY_TIME / DUE_TIME → fenêtres horaires en minutes depuis 00:00

0-BIS.2 Dataset 2 — Real-World VRP avec contraintes réelles
  Source    : https://data.4tu.nl/datasets/1e23b9d7-58ce-4567-a5e5-14b7cebbb3bc
  Auteur    : Emir Žunić (2019)
  Licence   : Open Data — usage libre avec citation
  Format    : Excel (.xls) — 9 fichiers
  Contenu   : Données GPS réelles d'une entreprise de distribution (Bosnie)

  Ce dataset contient :
    - Coordonnées GPS réelles (latitude/longitude directement utilisables)
    - Contraintes de capacité véhicule réelles
    - Fenêtres horaires issues d'opérations réelles
    - Solutions de référence pour valider tes algorithmes

  Instructions d'utilisation dans l'application :
    1. Convertir les fichiers .xls en CSV via pandas :
       df = pd.read_excel('fichier.xls')
       df.to_csv('fichier.csv', index=False)
    2. Mapper les colonnes vers le format interne CityPulse :
       GPS réel → latitude / longitude directement (pas de conversion nécessaire)
    3. Utiliser ce dataset pour la démonstration "mode réel"
       (contrairement à Solomon qui est "mode démo académique")

BLOC 3 — MOTEUR D'OPTIMISATION IA (CŒUR DU PROJET — 15 pts)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3.1 Algorithme 1 — Glouton (Nearest Neighbor)
  - Calcul matrice de distances haversine (GPS réels)
  - Construction de tournée greedy : client non visité le plus proche
  - Respect de la contrainte de capacité (kg et m³)
  - Multi-véhicules : distribution automatique
  - Sortie : liste ordonnée par véhicule + métriques
  - Temps d'exécution mesuré via time.perf_counter() (ms)

3.2 Algorithme 2 — 2-opt (Amélioration locale)
  - Initialisation depuis solution glouton ou solution aléatoire
  - Inversion de segments pour éliminer les croisements
  - Enregistrement de chaque amélioration (courbe de convergence)
  - Condition d'arrêt : no-improvement ou max_iterations (configurable)
  - Variante Or-opt disponible (déplacement d'un segment)
  - Gain % vs glouton affiché en temps réel
  - Temps d'exécution mesuré via time.perf_counter() (ms)

3.3 Algorithme 3 — OR-Tools Google (VRP Solver)
  - Modélisation VRP complète avec OR-Tools Routing Library
  - Contraintes : capacité kg, fenêtres temporelles, multi-dépôts
  - Stratégie initiale : PATH_CHEAPEST_ARC
  - Méta-heuristique : GUIDED_LOCAL_SEARCH
  - Limite de temps CPU configurable (défaut 30s, max 300s)
  - Extraction et formatage de la solution optimale
  - Temps d'exécution mesuré via time.perf_counter() (ms)

3.4 Tableau de comparaison des 3 algorithmes
  - Distance totale (km)
  - Durée totale estimée (h:mm) — incluant coefficient trafic × météo
  - Coût carburant total (€)
  - Taux de respect des fenêtres horaires (%)
  - Retard moyen par livraison (min)
  - Temps CPU de l'algorithme (ms) — via time.perf_counter()
  - Gain relatif vs algorithme glouton (%)
  - Taux d'utilisation moyen de la flotte (%)
  - Mise en évidence automatique de la meilleure solution (toutes métriques)

  Cibles de qualité (issues du CDC) :
    Distance    : OR-Tools < 2-opt < Glouton (toujours)
    Gain 2-opt  : entre 5% et 20% vs glouton
    Respect horaires : > 85% avec OR-Tools en conditions normales
    Temps CPU   : Glouton < 1s | 2-opt < 10s | OR-Tools < 30s

3.5 Configuration avancée du moteur IA
  - Sélection individuelle ou comparaison des 3
  - Paramètre : limite temps OR-Tools
  - Paramètre : iterations max 2-opt
  - Paramètre : priorité (minimiser distance / coût / retards)
  - Mode « Rapide » (glouton uniquement) vs mode « Optimal » (OR-Tools)
  - Lancement en arrière-plan (QThread non bloquant)
  - Annulation du calcul en cours

3.6 Fonctionnalités d'optimisation avancées
  - Optimisation en temps réel simulé : recalcul si un véhicule tombe en panne
  - Réaffectation dynamique : redistribuer les livraisons d'un véhicule indisponible
  - Optimisation partielle : re-optimiser une zone géographique uniquement
  - Contrainte de retour au dépôt ou fin en un point arbitraire
  - Support des livraisons avec ramassages simultanés (pickup & delivery)
  - Priorité client intégrée dans le coût de la solution

BLOC 4 — SIMULATION TRAFIC & MÉTÉO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  NOTE IMPORTANTE : Le trafic et la météo sont des paramètres de simulation
  choisis manuellement par l'utilisateur via l'interface. Aucune API externe
  n'est utilisée. Ce sont des coefficients multiplicateurs appliqués aux
  temps de trajet calculés par les algorithmes.

4.1 Simulation du trafic (sélection manuelle dans l'interface)
  - 4 niveaux prédéfinis : Fluide ×1.0 | Chargé ×1.3 | Heure de pointe ×1.7 | Accident ×2.2
  - Mode aléatoire : coefficient entre 1.0 et 2.5 (distribution uniforme)
  - Mode personnalisé : coefficient libre (1.0 → 5.0)
  - Trafic par zone géographique (North/South/Center)
  - Trafic par heure de la journée (courbe horaire configurable)
  - Simulation d'accident sur un arc spécifique (zone noire sur la carte)

4.2 Simulation météo (sélection manuelle dans l'interface)
  - 5 conditions : Ensoleillé ×1.0 | Pluie légère ×1.1 | Pluie forte ×1.25 | Neige ×1.6 | Verglas ×2.0
  - Combinaison automatique : coefficient final = trafic × météo
  - Icônes météo dans l'interface
  - Impact visuel sur la carte (overlay coloré selon condition)

4.3 Contraintes opérationnelles
  - Capacité poids (kg) par véhicule
  - Capacité volume (m³) par véhicule
  - Fenêtres horaires strictes par client (issues du dataset Solomon : READY_TIME / DUE_TIME)
  - Fenêtres horaires souples avec pénalité de retard
  - Priorité client (livraison prioritaire en premier)
  - Temps de service par client (SERVICE_TIME dans Solomon)
  - Pause obligatoire pour le chauffeur (ex : 30min après 4h de conduite)
  - Distance maximale par tournée (limiter le rayon d'action)
  - Nombre maximum de clients par véhicule (configurable)

BLOC 10 — INTELLIGENCE ARTIFICIELLE AVANCÉE (AU-DELÀ DU CDC)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

10.1 Prédiction de retards
  - Modèle ML (régression) pour prédire le retard probable d'une livraison
  - Entrées : heure de départ, coefficient trafic, coefficient météo, distance,
    historique du client (données accumulées en SQLite)
  - Sortie : retard estimé en minutes + intervalle de confiance
  - Affichage dans la feuille de route : badge "Risque de retard"
  - Données d'entraînement : historique des tournées sauvegardées en BDD

10.2 Recommandation intelligente de scénario
  - Suggère automatiquement la meilleure combinaison trafic/météo/algo
  - Basé sur l'historique des tournées passées (BDD SQLite)
  - Justification textuelle de la recommandation

10.3 Détection d'anomalies
  - Détecte les livraisons statistiquement anormales (coût, distance, retard)
  - Méthode : Z-score ou IsolationForest (scikit-learn) — pas d'entraînement requis
  - Alerte visuelle dans le dashboard
  - Rapport d'anomalies hebdomadaire

10.4 Clustering géographique des clients
  - Algorithme K-means (scikit-learn) sur les coordonnées GPS des clients
  - Pas d'entraînement requis — fonctionne directement sur les données chargées
  - Suggestion automatique de zones de tournée
  - Visualisation des clusters sur la carte

10.5 Prévision de la demande
  - Analyse des tendances de livraison sur les données historiques (BDD SQLite)
  - Prévision du nombre de livraisons sur les 7 prochains jours
  - Graphique de prévision dans le dashboard
